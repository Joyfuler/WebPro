개발 후 배포 단계 (분석-설계-개발-테스트 - 유지보수)
요구사항확인은 이 중 분석에 해당
1) 현행시스템 분석 및 개발기술 환경 정의
새로운 시스템을 처음부터 구축하는 경우는 적음. 대부분 이미 있는 시스템을 변경/업데이트하는 경우가 많으므로
현재 있는 시스템을 파악해야 함. - 향후 개발하려는 시스템의 이해도를 높이고, 개발 방향 설정에 도움.

확인해야 하는 부분으로는 
(1) 구성/기능/인터페이스. - 제공기능과 정보 교환방법 파악
(2) SW 구성 - 사용중인 SW와 기술요소 파악
운영체제 (윈도우 / 리눅스 / 맥 / 안드로이드 등)
DBMS (성능/기술지원,구축...) - 오라클 / MySql / MongoDB등...
오픈소스 (우편번호 찾기 등..)
(3) HW 구성 - 사용중인 HW와 네트워크 구성 파악
미들웨어 (톰캣 등) 웹어플리케이션 서버 (지속여부 확인)

2) 요구사항 정의ㅌ
도출-분석-명세-확인의 4단계 구성
1) 도출 - 개발자와 고객간 커뮤니케이션
2) 분석 - 요구사항 중 되는 것과 안 되는 것을 조율
3) 명세 - 요구사항에 대한 내용을 컨펌할 수 있도록 문서 작성
4) 확인 - 문서 검토 후 요구사항 이해했는지 여부 확인하고 컨펌. 문제가 있을 경우는 피드백

요구사항 확인 -> 명세 검증 ->  비용추정 -> 요구사항 확인.. 사이클이 가능하도록 정확하게 기술할 것

3) 유스케이스 모델

유스케이스 모델에는 기본적으로 모든 기능을 넣고, 겹치는 부분은 include 등으로 표현.
액터 (옷) - 이벤트 시작 객체, 기능 구현 관련 액터를 모두 도출.
유스케이스 (UseCase1) - 시스템에 대한 시나리오 집합 다른 유스케이스와 겹치는 기능이 없는지 확인.
명세서 - 주요 이벤트 흐름을 도출

ERM ( 각 데이터 한 줄(NTT)의 집합 및 관계 모델) - 이를 ERD로 문서화함.
